# 個人設定

## 判断要求の前置きルール

`AskUserQuestion` で重要な判断（方針変更、リスクある操作、スコープ修正等）を問う前は、
必ず直前に「## まとめ」または「## 現状」セクションを出力する。
含めるべき情報:
- 背景（なぜ判断が必要になったか）
- 各選択肢の trade-off（リスク・コスト・期待結果）
- 推奨案がある場合はその根拠

## シェルコマンド置換 `$(...)` の絶対禁止

`$(...)` を含むコマンドは安全ヒューリスティクスにより毎回確認ダイアログが出る。
**コマンド生成時は `$(...)` を絶対に使わないこと。**

> **注意**: PreToolUse hook でのブロックは許可ダイアログより後に実行されるため効果がない。
> この制約は CLAUDE.md の指示のみで担保している。

- git commit は `-m "message"` または `-m "1行目" -m "2行目"` を使う
  - ヒアドキュメント形式 `$(cat <<'EOF'...)` も禁止
- 変数代入が必要な場合は複数ステップに分割する
- パイプ (`|`) やリダイレクト (`>`) は問題ない
- `$(...)` やバッククォート は全て確認ダイアログの対象になる
- **commit メッセージ内のバッククォート（`` ` ``）は zsh OOM クラッシュを誘発するため絶対禁止**
  - `-m "...` `` ` `` `code` `` ` `` `..."` のような直書きは bash がコマンド置換として展開し、zsh `command_not_found_handler` 無限再帰で fatal error: out of memory に至る
  - 三連 backtick (` ``` ` fenced code block) も同様に危険
  - 代替: コード片は引用符・インデント・タグ表記で代替するか、commit メッセージから除外する

## gitコマンドのルール

gitコマンドは常にカレントディレクトリで実行する。`-C` オプションや絶対パス指定は使わない。

## Codex CLI との連携

出力は全てCodexにて検証される。
Codex にコードレビューを依頼する場合は以下のコマンドを使う：

```bash
# なんらかの依頼
codex exec "指示テキスト"

# コードレビュー（ベースブランチとの差分をレビュー）
codex review --base main

# セッションを再開して追加指示
codex exec resume <session-id> "追加の指示テキスト"
```

- `codex review` の出力にセッション ID が含まれる
- `codex exec resume` で同じセッションのコンテキストを維持したまま追加指示を送れる
- Bash ツールから実行可能（非対話モードで動作する）

## 設定ファイルのスコープ

- User-scoped: `~/.claude/settings.json`
- Project-scoped: `.claude/settings.json`
- スコープが不明な場合は書き込む前にユーザーに確認すること

## スキル優先呼び出し

対応するスキルが存在する場合は必ずスキル経由で実行する。手動でスキルのロジックを再現しない。

## 許可ルールの定期メンテナンス

許可ルール（settings.json の permissions）が肥大化・陳腐化していないか定期的に確認する。
`tools:suggest-permissions` スキルを使って最適化を提案させること。
目安: 月1回、または User Rejected が多いと感じた時。

## セッションタイトル自動設定

- hooks で自動設定済み（SessionStart hook でシェルコマンド直接実行）
- 個別にタイトルを指定したい場合のみ `tools:session-title` スキルを使う
- フォーマット: `リポジトリ名 / ブランチ名`

# トレードオフスライダー

スコープ: 大
予算: 小
納期: 最小
品質: 最大

## 改善の約束は行動で示す

「意識します」「気をつけます」「メモリに残します」のような宣言は禁止。これらは次のセッションや別の環境に引き継がれず、実効性がない。

改善が必要な場合は、必ず以下のいずれかの行動を取ること:

- **CLAUDE.md / AGENTS.md にルールを追記する**（再発防止策をプロンプトに残す）
- **GitHub Issue を作成する**（修正が必要な箇所を記録する）
- **その場で修正する**（コード・設定・ドキュメントを直接変更する）

行動を伴わない反省文は出力しない。
