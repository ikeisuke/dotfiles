### 改善の約束は行動で示す

「意識します」「気をつけます」「メモリに残します」のような宣言は禁止。これらは次のセッションや別の環境に引き継がれず、実効性がない。

改善が必要な場合は、必ず以下のいずれかの行動を取ること:

- **CLAUDE.md / AGENTS.md にルールを追記する**（再発防止策をプロンプトに残す）
- **GitHub Issue を作成する**（修正が必要な箇所を記録する）
- **その場で修正する**（コード・設定・ドキュメントを直接変更する）

行動を伴わない反省文は出力しない。

## コマンド実行ルール

### シェルコマンド置換 `$(...)` の絶対禁止

`$(...)` を含むコマンドは安全ヒューリスティクスにより毎回確認ダイアログが出る。
**コマンド生成時は `$(...)` を絶対に使わないこと。**

> **注意**: PreToolUse hook でのブロックは許可ダイアログより後に実行されるため効果がない。
> この制約は CLAUDE.md の指示のみで担保している。

- git commit は `-m "message"` または `-m "1行目" -m "2行目"` を使う
  - ヒアドキュメント形式 `$(cat <<'EOF'...)` も禁止
- 変数代入が必要な場合は複数ステップに分割する
- パイプ (`|`) やリダイレクト (`>`) は問題ない
- `$(...)` やバッククォート は全て確認ダイアログの対象になる
- **commit メッセージ内のバッククォート（`` ` ``）は zsh OOM クラッシュを誘発するため絶対禁止**
  - `-m "...` `` ` `` `code` `` ` `` `..."` のような直書きは bash がコマンド置換として展開し、zsh `command_not_found_handler` 無限再帰で fatal error: out of memory に至る
  - 三連 backtick (` ``` ` fenced code block) も同様に危険
  - 代替: コード片は引用符・インデント・タグ表記で代替するか、commit メッセージから除外する

### 外部スクリプト実行時のパイプの扱い

ファイル書き出しやログ生成を伴う外部スクリプトを実行する際、その出力に**バッファリング/切り詰めパイプを直接挟まない**。理由:

- 出力が見えなくなり、事後にエラー詳細やログを参照できなくなる
- `head` 等は SIGPIPE で上流プロセスを早期終了させる場合がある

**禁止例**:
- `script | head -N`
- `script | tail -N`
- `script > /dev/null`

**許容パターン**: `tee` で全出力をファイルに残してから表示を絞る。

- `script | tee /tmp/out.log | tail -n 50` — 全出力を `/tmp/out.log` に保存しつつ表示だけ tail に絞る
- 後で `cat /tmp/out.log` や `grep` で全文参照可能
- 標準エラーも残したい場合は `script 2>&1 | tee /tmp/out.log | tail -n 50`

## git 運用

### カレントディレクトリで実行する

gitコマンドは常にカレントディレクトリで実行する。`-C` オプションや絶対パス指定は使わない。

### force push の AI 代行実行

`git push --force-with-lease` 等の force push 系操作を AI が実行する前に、AskUserQuestion で「AI 代行 / 中断 / 手動実行」の 3 択を提示すること。「AI 代行」が選択された場合のみ AI が実行できる。デフォルトで「手動実行」前提とせず、明示的に許可を取る。

特にリモートとローカルが分岐している（diverged）状態での force push はデータ消失リスクが高いため、push 前に `git status` 等で状態を確認する。

### 例外: squash 直後の force push は確認不要

以下の **すべて** を満たす場合、AskUserQuestion を経由せずに `git push --force-with-lease` を AI が直接実行してよい:

1. 直前に squash（`/squash-unit` スキルまたは `squash-unit.sh`）を実行し、`squash:success:<hash>` を確認している
2. `--force-with-lease`（または `--force-with-lease=<branch>:<expected-sha>`）を使用する（plain `--force` は対象外）
3. push 対象ブランチが `cycle/*` / `unit/*` / その他作業ブランチ（main / master / リリースブランチ系統には適用しない）

上記すべてを満たす場合のみ確認をスキップ可能。逸脱（plain `--force` を使う場合・main 系ブランチへの push・squash 由来でない force push）は従来通り 3 択提示が必須。

push 前に `git status` でツリーが clean であることと、ローカル commit が squash 結果の単一 commit であることを確認すること。

## エージェント連携

### Codex CLI との連携

出力は全てCodexにて検証される。
Codex にコードレビューを依頼する場合は以下のコマンドを使う：

```bash
# なんらかの依頼
codex exec "指示テキスト"

# コードレビュー（ベースブランチとの差分をレビュー）
codex review --base main

# セッションを再開して追加指示
codex exec resume <session-id> "追加の指示テキスト"
```

- `codex review` の出力にセッション ID が含まれる
- `codex exec resume` で同じセッションのコンテキストを維持したまま追加指示を送れる
- Bash ツールから実行可能（非対話モードで動作する）

目安: 月1回、または User Rejected が多いと感じた時。
